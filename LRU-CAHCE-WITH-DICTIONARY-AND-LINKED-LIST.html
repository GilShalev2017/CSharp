LRU Cache Explanation

Why both a Dictionary AND a LinkedList?

Each structure solves a different problem — you need both together:

Structure Purpose Operation
Dictionary Look up a node by key O(1)
LinkedList Track usage order (most → least recent) O(1) move to front

The key insight: The dictionary stores direct references to the LinkedListNode, not just the value. This is the clever part.

  • Without the dictionary: to find a key in the linked list you’d have to scan it — O(n)
  • Without the linked list: you’d have no way to efficiently track which item is “least recently used”
  • Together: you can find any node instantly via the dictionary, then move it in the linked list in O(1)

Here’s what memory looks like after Set("a",1), Set("b",2), Set("c",3):

Dictionary:          LinkedList (most → least recent):
"c" → [ref] ──────► [c,3] ↔ [b,2] ↔ [a,1]
"b" → [ref] ──────────────────►↗
"a" → [ref] ──────────────────────────►↗

What is return default(TValue)?

It returns the default zero-value for whatever type TValue is — it’s how C# handles “nothing to return” in a generic method.

TValue type default(TValue) returns
int 0
bool false
string null
object null
A struct All fields zeroed

This is why in the example cache.Get("a") after eviction prints 0 — the cache is LRUCache<string, int>, so default(int) is 0, signaling “key not found.” It’s essentially the generic equivalent of returning null, but safe for value types like int which can’t be null.

In modern C# you can also write just default instead of default(TValue) — they’re identical.
